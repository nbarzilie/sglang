# Benchmark Log Workflow and Analysis

This repository contains a small, purpose-built workflow for turning raw SGLang benchmark logs into clean, analyzable benchmark results. The workflow is implemented in three Python scripts under `scripts/`: `clean_bench_logs.py`, `cleaned_to_csv.py`, and `plot_bench_comparison.py`. Together they define a pipeline that starts with raw `.log` files in `input/`, extracts complete benchmark runs into normalized cleaned logs in `cleaned_input/`, converts those cleaned logs into a single CSV file in `csv/benchmark_results.csv`, and finally renders visual comparisons into `result_png/`.

The workflow is designed around a specific benchmark naming convention and a specific log structure. Each raw log filename encodes the backend, parallelism configuration, and dataset. A valid filename looks like `nixl_Ptp2_Dtp4_Pdp1_Ddp1_rand.log` or `mooncake_Ptp2_Dtp4_Pdp1_Ddp1_sharegpt.log`. The backend must be either `nixl` or `mooncake`. The parallelism fields are prefill tensor parallelism, decode tensor parallelism, prefill data parallelism, and decode data parallelism. The dataset must be one of `rand`, `sharegpt`, or `radixcache`. This naming convention matters because later analysis uses the filename as structured metadata, not merely as a display label.

The first stage is log cleaning. `clean_bench_logs.py` reads raw logs from `input/` and writes cleaned logs to `cleaned_input/`. It filters input files with a regular expression, so unexpected filenames are skipped with a clear message instead of being silently included. For every valid input log, the script looks for benchmark sections organized by a prompt-count header, a run header, optional token-count lines, and a benchmark result block. The important markers are `========== num-prompts:`, `--- Run`, `#Input tokens:`, `#Output tokens:`, `============ Serving Benchmark Result ============`, and the terminating line of equal signs.

The cleaner is intentionally conservative. It only writes a block when the block has a prompt header, run header, input token count, output token count, and a complete benchmark result section. If a run is interrupted, malformed, or missing required context, the script records a warning and skips that incomplete block. This prevents partially written log fragments from leaking into the final CSV and distorting performance analysis. The script also includes a fallback for token counts: if explicit `#Input tokens:` or `#Output tokens:` lines are missing, it tries to recover them from `Total input tokens` and `Total generated tokens` inside the benchmark result section. This makes the cleaning stage robust to slightly different log formats while still enforcing complete benchmark records.

The cleaned log format preserves the logical grouping of the original experiment. Each prompt-count section appears once, followed by one or more run blocks. Every run block contains the token counts and the serving benchmark result text. Blank lines separate runs for readability. The cleaned file name is derived from the raw file name by appending `_cleaned`, for example `mooncake_Ptp2_Dtp4_Pdp1_Ddp1_rand_cleaned.log`. At this point, the data is still text, but the noisy parts of the original logs have been removed and all retained blocks are complete enough for structured parsing.

The second stage is CSV conversion. `cleaned_to_csv.py` reads cleaned logs from `cleaned_input/` and writes one consolidated CSV file to `csv/benchmark_results.csv`. This script again depends on the file naming convention. It maps each cleaned file back to its original raw log name, validates that original name against the expected backend, parallelism, and dataset pattern, then extracts metadata from it. The metadata includes `backend`, `case`, `prefill_tp`, `decode_tp`, `prefill_dp`, `decode_dp`, `dataset`, `dataset_name`, and `original_log`.

The `case` field is a compact identifier for the parallelism configuration, such as `Ptp2_Dtp4_Pdp1_Ddp1`. The `dataset_name` field expands short dataset identifiers into more readable names: `rand` becomes `random`, `sharegpt` remains `sharegpt`, and `radixcache` becomes `generated-shared-prefix`. This gives the later plots enough context to be understandable without requiring the reader to decode file names manually.

The converter also tries to identify the model used by reading the corresponding raw input log. It searches for either a single-quoted or double-quoted `model=` value. If the raw file is missing or the model cannot be found, it records `unknown`. This model extraction is useful because the cleaned logs mainly contain measured serving results, while the model name may only appear in the command or configuration text near the top of the raw log.

Inside each cleaned log, `cleaned_to_csv.py` parses prompt headers and run headers. A prompt header establishes the active `num_prompts` value. A run header starts a new row. Every following colon-separated line is treated as a metric or attribute for that row until another prompt header or run header appears. Labels are normalized by removing a leading `#`, lowercasing, replacing non-alphanumeric sequences with underscores, and trimming underscores. For example, `Mean TTFT (ms)` becomes `mean_ttft_ms`, and `#Input tokens` becomes `input_tokens`.

Values are parsed into useful Python types before being written to CSV. Integer-looking values become integers, numeric values with decimals become floats, `inf` becomes mathematical infinity, and `not set` stays as text. Other values remain strings. This is a practical parsing strategy for benchmark logs because the result block mixes numeric latency metrics, throughput metrics, backend names, and configuration strings. One special case is the `Backend` metric inside the serving result block: since `backend` is already used for the experiment backend from the filename, the serving result backend is written as `serving_backend` to avoid overwriting the higher-level comparison key.

The CSV writer uses a preferred column order for the fields that are most important to analysis. It starts with experiment identity and metadata, then workload dimensions, then the core benchmark metrics. Any additional parsed fields are appended afterward in sorted order. This means the CSV remains stable and readable for common analysis while still preserving extra metrics that may appear in future benchmark output. In the current data, the resulting CSV contains per-run rows with fields such as `num_prompts`, `input_tokens`, `output_tokens`, `successful_requests`, `benchmark_duration_s`, `mean_ttft_ms`, `median_ttft_ms`, `p99_ttft_ms`, `mean_itl_ms`, `median_itl_ms`, `p95_itl_ms`, `p99_itl_ms`, request throughput, token throughput, and end-to-end latency metrics.

The third stage is plot generation. `plot_bench_comparison.py` reads `csv/benchmark_results.csv` and creates comparison plots in `result_png/`. It imports `pandas` and `matplotlib` lazily through `load_dependencies()`, setting `MPLCONFIGDIR` to a repository-local `.matplotlib-cache` directory and forcing Matplotlib to use the non-interactive `Agg` backend. That makes the plotting script suitable for headless environments where a graphical display is unavailable.

Before plotting, the script validates that the input CSV has all required columns. The required set includes identity fields such as `backend`, `case`, `dataset`, `dataset_name`, and `model`; parallelism fields; the selected x-axis column; the run number; and the two primary metrics, `mean_ttft_ms` and `mean_itl_ms`. The x-axis can be either `input_tokens` or `num_prompts`, selected with `--x`. The default is `input_tokens`.

The plotting script groups the CSV by `case` and `dataset`. For each group, it checks that both `nixl` and `mooncake` are present. If either backend is missing, that group is skipped. This check is important because the script is explicitly producing backend comparison plots. A single-backend plot could be useful in another context, but it would not answer the main question this workflow is built around.

For each valid group, the script aggregates rows by backend and workload x-axis value. It computes the mean `mean_ttft_ms`, mean `mean_itl_ms`, and a count of runs. The use of a mean across runs reduces noise when multiple repeated measurements exist for the same backend and workload. It then creates a two-panel figure. The top panel plots mean time to first token, or TTFT, against the workload dimension. The bottom panel plots mean inter-token latency, or ITL, against the same workload dimension. Both panels use one line per backend, with markers and a legend.

The x-axis uses a base-2 logarithmic scale. This matches the workload pattern in the available benchmark data, where prompt counts often progress through powers of two such as 1, 2, 4, 8, and so on. A log scale makes it easier to compare behavior across small and large workloads in the same figure. Without that scale, the largest workload values would dominate the horizontal space and make the low-workload region difficult to inspect.

The generated plot title identifies the comparison as NIXL versus Mooncake and includes metadata derived from the CSV. The metadata includes the dataset, readable dataset name, model, and parallelism configuration. The output filename includes the case, dataset, and x-axis, for example `Ptp2_Dtp4_Pdp1_Ddp1_rand_input_tokens_comparison.png`. The image metadata also includes a title and description, which makes the generated PNG files more self-describing when inspected outside the repository.

The analysis performed by this workflow is primarily a backend performance comparison. It compares NIXL and Mooncake under the same model, dataset, parallelism case, and workload dimension. The main latency metrics are TTFT and ITL. TTFT measures how long it takes before the first generated token is returned, so it is sensitive to prefill cost, queueing, scheduling, and startup overhead for a request batch. ITL measures the average time between generated tokens after generation begins, so it reflects decode efficiency and sustained serving behavior. Looking at both metrics is necessary because a backend can have strong decode throughput but poor first-token latency, or the opposite.

The CSV also enables throughput analysis. Fields such as `request_throughput_req_s`, `output_token_throughput_tok_s`, `input_token_throughput_tok_s`, and `total_token_throughput_tok_s` show how much work the serving system completed per second. These metrics complement latency. Low latency at tiny workloads is useful, but for batch-heavy serving the important question is often whether throughput scales as prompt count or input token volume increases. The existing summary CSV files show this kind of aggregation by dataset, backend, and workload.

The dataset dimension is central to interpreting the results. The `rand` dataset represents random prompts, `sharegpt` represents conversational ShareGPT-style inputs, and `radixcache` represents generated shared-prefix workloads. These workloads stress different parts of the system. Random prompts are useful for general scaling behavior. ShareGPT-like prompts are closer to conversational serving. Shared-prefix workloads are especially relevant when caching behavior, prefix reuse, or large repeated context matters. Because the scripts keep dataset and dataset name in every row, the same analysis can be repeated cleanly across all workload families.

The workflow also exposes data quality issues instead of hiding them. Invalid filenames are skipped. Incomplete cleaned blocks are warned about. Missing comparison backends cause a plot group to be skipped. Missing required CSV columns stop plotting with an explicit error. This is a good pattern for benchmark analysis because silent inclusion of bad data can produce convincing but false charts. The scripts prefer deterministic, explainable filtering.

The current repository contents show the pipeline has already been run. There are six cleaned logs for `Ptp2_Dtp4_Pdp1_Ddp1` across two backends and three datasets. The consolidated benchmark CSV has per-run results, and the `result_png/` directory contains comparison plots for both `input_tokens` and `num_prompts`. There are also summary CSV files by dataset/backend and workload/backend, which appear to be downstream analysis products derived from the main benchmark CSV.

A typical run order is straightforward. First, run `scripts/clean_bench_logs.py` to regenerate `cleaned_input/` from `input/`. Second, run `scripts/cleaned_to_csv.py` to regenerate `csv/benchmark_results.csv`. Third, run `scripts/plot_bench_comparison.py --x input_tokens` and optionally `scripts/plot_bench_comparison.py --x num_prompts` to refresh plots. After that, inspect the CSV summaries and PNG charts to compare backend latency and throughput by dataset and workload.

There are also a few interpretation limits to keep in mind. The scripts compute simple means across repeated runs, but they do not calculate confidence intervals, standard deviations, or outlier filters. That means the plots are best read as direct operational summaries rather than a full statistical study. The workflow also assumes that rows sharing the same case, dataset, model, and workload are comparable. If hardware, server settings, request generation behavior, or cache state changed between raw logs, that context needs to be controlled outside these scripts or added as extra metadata. The current analysis is strongest when the raw logs were produced in a disciplined benchmark run where only the backend changes between comparable measurements. In that setting, the pipeline gives a clear, repeatable view of latency and throughput behavior.

In short, the workflow converts raw benchmark text into structured evidence. The cleaning stage protects the analysis from partial logs. The CSV conversion stage turns repeated benchmark runs into machine-readable rows with consistent metadata. The plotting stage aggregates repeated runs and visualizes the two most important latency metrics across workload sizes. The resulting analysis is focused on whether NIXL or Mooncake performs better for a given dataset, model, parallelism configuration, and workload scale.
